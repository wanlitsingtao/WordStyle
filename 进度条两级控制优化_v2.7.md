# 进度条两级控制优化说明 - v2.7

## 🎯 优化目标

实现更精确的进度条显示，让用户清楚了解整体转换进度和当前文件的处理状态。

## ❌ 优化前的问题

### 问题描述

**单文件模式**：
- 进度条不更新或更新不准确
- 用户无法知道当前处理到哪一步

**多文件模式**：
- 进度条只显示当前文件的内部进度
- 无法反映整体转换进度
- 例如：3个文件，处理第2个文件的第5步时，进度条显示 71%（5/7）
- 但实际整体进度应该是：(1 + 5/7) / 3 = 57%

### 用户体验问题

```
场景：转换3个文件，每个文件7个步骤

当前行为：
文件1: 进度 0% → 14% → 28% → ... → 100%
文件2: 进度 0% → 14% → 28% → ... → 100%  ← 看起来像是重新开始
文件3: 进度 0% → 14% → 28% → ... → 100%  ← 又重新开始

用户困惑：
"怎么进度条一直在重置？到底完成了多少？"
```

## ✅ 优化后的方案

### 两级进度控制策略

#### 第一级：文件级别（宏观进度）

```
总文件数: n
每个文件占: 1/n 的进度（即 100/n 个单位）

示例（3个文件）：
文件1范围: 0% - 33.3%
文件2范围: 33.3% - 66.7%
文件3范围: 66.7% - 100%
```

#### 第二级：单文件内部（微观进度）

```
单个文件的处理步骤: 7步
每步占当前文件范围的: 1/7

示例（3个文件，第2个文件的第4步）：
基础进度: 33.3%（文件1已完成）
当前步进度: (4/7) × 33.3% = 19.0%
总进度: 33.3% + 19.0% = 52.3%
```

### 计算公式

```python
# 总体进度 = 已完成文件的进度 + 当前文件内部的进度
total_progress = (completed_files × 100) + (current_step / total_steps × 100)

其中：
- completed_files: 已完成的文件数（0, 1, 2, ...）
- current_step: 当前步骤（1-7）
- total_steps: 总步骤数（固定为7）

进度条最大值 = 总文件数 × 100
```

### 示例计算

**场景**：3个文件，正在处理第2个文件的第4步

```python
completed_files = 1  # 文件1已完成
current_step = 4     # 当前是第4步
total_steps = 7      # 总共7步

file_progress = (4 / 7) × 100 = 57.14
total_progress = (1 × 100) + 57.14 = 157.14

进度条显示: 157.14 / 300 = 52.38%
```

## 🔧 技术实现

### 核心改进点

#### 1. 初始化进度条最大值

**单文件模式**：
```python
# 在开始转换时
total_files = 1
self.loading_progress['maximum'] = total_files * 100  # 最大值 = 100
self.loading_progress['value'] = 0
```

**多文件模式**：
```python
# 在开始转换时
total_files = len(self.source_files)
self.loading_progress['maximum'] = total_files * 100  # 例如 3个文件 = 300
self.loading_progress['value'] = 0
```

#### 2. 单文件模式的进度回调

```python
def progress_callback(step, message):
    def update_ui():
        self.log(f"  [{step}/7] {message}")
        
        # 更新进度条（两级进度）
        # 第一级：文件级别（已完成0个文件）
        # 第二级：单文件内部（当前步骤/总步骤）
        file_progress = (step / 7) * 100  # 当前文件的进度（0-100）
        total_progress = file_progress     # 单文件模式，直接使用该进度
        self.loading_progress['value'] = total_progress
        self.root.update_idletasks()
    
    self.root.after(0, update_ui)
```

#### 3. 多文件模式的进度回调

```python
def progress_callback(step, message):
    def update_ui():
        self.log(f"    [{step}/7] {message}")
        
        # 更新进度条（两级进度）
        # 第一级：文件级别（已完成 idx-1 个文件）
        # 第二级：单文件内部（当前步骤/总步骤）
        completed_files = idx - 1          # 已完成的文件数
        file_progress = (step / 7) * 100   # 当前文件的进度（0-100）
        total_progress = (completed_files * 100) + file_progress
        self.loading_progress['value'] = total_progress
        self.root.update_idletasks()
    
    self.root.after(0, update_ui)
```

### 关键约束

✅ **进度不会超过当前文件的上限**
- 文件1未完成时：progress < 100
- 文件2未完成时：progress < 200
- 文件3未完成时：progress < 300

✅ **进度单调递增**
- 不会出现回退
- 不会出现跳跃

✅ **精确反映整体进度**
- 用户可以清楚看到完成了多少百分比

## ⚠️ 注意事项

### 1. 进度条最大值设置

**原理**：
```python
# 不是设置为 100，而是设置为 总文件数 × 100
self.loading_progress['maximum'] = total_files * 100
```

**原因**：
- Tkinter Progressbar 的 value 可以是任意数值
- 通过设置更大的 maximum，可以实现更精细的控制
- 实际显示时会自动归一化到 0-100%

### 2. 步骤数固定为 7

**当前实现**：
```python
file_progress = (step / 7) * 100
```

**假设**：
- `full_convert()` 有 7 个主要步骤
- 如果未来步骤数变化，需要修改这个常量

**建议**：
- 可以将 7 定义为常量：`TOTAL_STEPS = 7`
- 或者从 `full_convert()` 动态获取步骤数

### 3. 线程安全

**使用 `root.after()`**：
```python
self.root.after(0, update_ui)
```

**原因**：
- 转换在后台线程执行
- GUI 更新必须在主线程执行
- `after(0)` 确保线程安全
